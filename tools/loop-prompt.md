# StudyHub 전수 개선 루프 프롬프트

> 사용법: `/loop 20m @tools/loop-prompt.md`

---

## ⛔ 절대 규칙 — 반드시 읽고 시작

아래 두 스킬은 **선택이 아니라 의무**다. 호출만 하고 무시하는 것은 금지다.
스킬이 정의한 프로세스를 **끝까지 실제로 따른 후** 구현으로 넘어간다.

1. **`creative-brainstorming` 스킬** — 매 루프 시작 시 반드시 호출
   - 스킬 내 **발산(아이디어 50개 이상) → 수렴(상위 10개 선별) → 검증** 프로세스를 **실제로 전부 실행**
   - 아이디어 목록 없이 구현으로 넘어가는 것 금지
   - 스킬이 요구하는 기법(SCAMPER, 역발상, Worst Idea 등) 중 최소 3가지 사용

2. **`frontend-design` 스킬** — 시각·레이아웃·CSS 변경이 포함되면 반드시 호출
   - 스킬이 정의한 디자인 원칙과 컴포넌트 가이드라인을 따라 HTML/CSS 설계
   - 스킬 없이 임의로 UI를 변경하는 것 금지

---

## 실행 지시

너는 StudyHub(`F:\work\studyhub`)의 **전수 개선 에이전트**다.
매 실행마다 `tools/loop-state.json`에서 `angleIndex`를 읽어 해당 관점으로
**한국어(ko/) 전체 페이지를 전수 감사**하고 발견한 문제를 즉시 고쳐라.
항상 **한국어(ko/) 먼저** 수정하고, 이후 en/ → ja/ → zh/ 순으로 전파한다.

## 8개 순환 감사 관점

| 인덱스 | 관점 | 핵심 질문 | 주요 점검 대상 |
|--------|------|-----------|----------------|
| **0** | 콘텐츠 깊이 | 공식만 있나 원리·WHY까지 설명하나? | ko/ 전체 article |
| **1** | 첫인상 설득력 | 3초 안에 "여기 있어야겠다"를 느끼는가? | ko/index.html, ko/*/index.html |
| **2** | 학습 효과 장치 | 연습문제·퀴즈·암기팁·단계별 예제가 충분한가? | ko/ 전체 article |
| **3** | 신뢰 신호 | 교육과정 근거·오류 없음·정확성 표시가 있는가? | ko/ 전체, 특히 수학·과학 |
| **4** | 차별화 가치 | 네이버/EBS에서 찾을 수 없는 독자적 가치가 있는가? | ko/ 전체 |
| **5** | 리텐션 훅 | 읽고 나서 다음 페이지로 자연스럽게 이어지는가? | 모든 article 하단 |
| **6** | 정보 구조 | 원하는 내용을 30초 안에 찾을 수 있는가? | 인덱스·nav·breadcrumb |
| **7** | 모바일 UX | 스마트폰에서 표·수식이 망가지지 않는가? | CSS, 테이블·수식 |

## 상세 실행 절차

### STEP 1 — 상태 파일 읽기
`tools/loop-state.json` 읽기 → `angleIndex` 확인 (없으면 0)

---

### STEP 2 — creative-brainstorming 스킬 실행 ⛔ 반드시

`creative-brainstorming` 스킬을 호출하고 스킬이 정의한 프로세스를 **전부 실행**:

**2-1. 발산 단계** (아이디어 50개 이상, 판단 금지):
- SCAMPER 기법으로 현재 관점에서 StudyHub의 각 요소를 7가지 방향으로 비틀기
- 역발상: "당연히 교육 사이트는 X다" → "만약 X가 아니라면?" 질문
- Worst Idea: "절대 안 되는 아이디어"를 먼저 나열하고 뒤집기
- Forced Connection: 무관한 분야(게임, SNS, 뉴스, 쇼핑몰)와 강제 연결
- 아이디어 번호를 붙여 목록으로 나열 (1~50번 이상)

**2-2. 수렴 단계** (상위 10개 선별):
- "어?!" 반응이 나오는 아이디어 기준으로 선별
- 각 아이디어를 한 단락으로 확장 설명

**2-3. 검증 단계**:
- 이미 사이트에 존재하는가?
- 왜 재밌거나 효과적인가?
- 이번 루프 안에 구현 가능한가?
- 최종 구현 대상 3~5개 확정

---

### STEP 3 — frontend-design 스킬 실행 (시각 변경 포함 시 ⛔ 반드시)

STEP 2에서 선별된 아이디어 중 시각·레이아웃·CSS 변경이 포함되면:
`frontend-design` 스킬을 호출하고 스킬의 디자인 원칙에 따라 구체적인 HTML/CSS 설계 수립

---

### STEP 4 — 한국어 전수 감사

`F:\work\studyhub\ko\` 디렉토리의 **모든 HTML 파일**을 읽고
현재 관점 기준으로 문제점 목록화:
- 파일명, 문제 유형, 심각도(critical / high / medium)

---

### STEP 5 — 한국어 즉시 수정

STEP 2에서 확정된 아이디어 + STEP 4 감사 결과를 ko/ 파일에 전부 적용:
- critical → 반드시 이번 루프에서 수정
- high → 가능한 한 수정
- 기존 CSS 클래스(`.why-box`, `.misconception`, `.tldr`, `.concept-box`, `.tip`, `.example` 등) 최대한 재사용
- 새 컴포넌트가 필요하면 `css/style.css`에 추가 후 전 언어 적용

---

### STEP 6 — 다국어 전파

ko/ 변경 내용을 en/ → ja/ → zh/ 순으로 동기화:
- 구조·레이아웃 변경 → 4개 언어 모두 동기화
- 교과 고유 내용 → 해당 언어의 동등한 수준 콘텐츠로 추가

---

### STEP 7 — 상태 저장 및 커밋

`tools/loop-state.json` 업데이트:
```json
{
  "angleIndex": (이전값 + 1) % 8,
  "lastRun": "ISO 날짜",
  "lastAngle": "관점 이름",
  "filesChanged": 수정된 파일 수,
  "summary": "이번 루프에서 한 일 한 줄 요약"
}
```

Git 커밋:
```
git add ko/ en/ ja/ zh/ css/ tools/loop-state.json
git commit -m "루프[관점명]: [요약]"
```

## 주의사항

- **한국어 먼저** — ko/ 수정 완료 후 다국어 전파. 절대 순서 바꾸지 말 것
- **스킬 프로세스는 생략 불가** — 호출만 하고 무시하는 것은 이 루프의 존재 이유를 부정하는 것
- CSS는 `css/style.css`에 중앙화. 개별 파일에 inline style 추가 금지
